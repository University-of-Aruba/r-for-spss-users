01 pedagogy · expect from educators

How do I stop students from just using AI?

Fundamentals are not an alternative to using AI — they are what makes AI use effective. A student who cannot read R cannot tell you whether Claude's output is right or wrong.

The question to examine is not how to prevent AI use. It is whether your assessment tests understanding or tests procedure. If a student can fully outsource an assignment to an LLM and pass, the assignment was probably testing the wrong thing regardless of AI.

What works better than surveillance: short oral defences of submitted code (five minutes is enough), process documentation where students describe what they tried and why, and locally specific datasets — Aruban census, island election results, local government data — that the model was not trained on and cannot bluff through.

If it keeps coming up, point to DCDC's Island Research Reference Data repo as a source of locally specific datasets.

02 privacy · expect from civil servants

Is it safe to use AI with government data?

The risk lives in what you put in the prompt, not where the code runs. R on your laptop never sends your data anywhere. Claude seeing your prompt is a separate question.

Practical rule of thumb. Code-generation prompts with no real data in them are low risk. Prompts that contain actual rows, names, IDs, or identifying fields are a privacy exposure regardless of where the script then runs.

If the data is genuinely sensitive, three safe options. Work with synthetic or anonymised data when asking the AI for help. Use a local LLM (Ollama running Llama or Mistral) so nothing leaves your machine. Or get an enterprise agreement with explicit no-training-on-inputs terms through your IT department.

The tool is not unsafe. The workflow is what matters. Keep real data out of the chat window.

03 the scariest question

If AI writes the code, why am I learning R?

Because you are the one standing in the meeting when someone asks what drove the number. The model can produce output. Only you can defend it.

This is the refinery question from this morning, in a new form. AI makes code faster to write. It does not make analysis easier to defend. If you cannot read the code the model produced, you cannot tell whether it answered the question you actually asked.

The analysts who get the most out of AI tools are the ones who know enough to double-check them. R literacy is what makes Claude or Copilot safe to use. It is also what lets you catch a subtle error before it ends up in a policy brief with your name on it.

Honest position: my own R code is AI-assisted. I could not do that if I could not read R.

04 the defensive question

Why R? SPSS works fine for me.

SPSS works fine until someone asks you to redo last quarter's analysis and you cannot remember which menu clicks you did. R remembers, because the script is the record.

This is not about SPSS being wrong. It is about what is written down. Menu clicks leave no trace. A script is a permanent, auditable record of what you did, readable by you next year and by a colleague who was never in the room. That is the core of reproducibility, and it is a DCDC priority for a reason — institutional memory survives staff turnover only if the work is written, not clicked.

R is also free, which matters for small institutions, and it has a much larger library of methods than SPSS. You do not have to choose one forever. Most people in this room will end up using both.

05 the practical question

What happens to all my .sav files?

They still open. R reads SPSS files directly through the haven package. You do not have to migrate anything to start.

One line: data <- haven::read_sav("myfile.sav"). Your variable labels, value labels, and missing value codes come along. Nothing is lost in translation, and you can keep SPSS open next to RStudio for as long as you need the comfort of the familiar interface.

The longer-term move is toward CSV as the storage format, because it is open, universal, and version-control friendly. But that is a choice for later, not a prerequisite for today.

Show the haven one-liner in Episode 2 or Episode 3 when the question actually comes up.

06 classification · expect from civil servants

We code ISIC and ISCO in SPSS syntax. How do we do that in R?

An R script is the direct equivalent of an SPSS syntax file — only auditable in git. And for ISIC/ISCO specifically, the R pattern is not a long chain of recodes: it is a join to a lookup table, which is simpler and less error-prone.

In SPSS, classification coding usually means a long RECODE or DO IF block in a .sps file, maintained by hand. The equivalent R script does the same work more explicitly: you hold your labour dataset in one tibble and the ISIC or ISCO code list in another, and you join them with dplyr::left_join(). One line. Every code gets its description attached, and any code that doesn't match shows up as NA — which is exactly the error you want to catch early.

The canonical ISIC or ISCO list itself is just a CSV file. Keep one copy on your network drive, SharePoint, or wherever your team already shares documents. Every analyst reads from the same file, so the codes stay consistent across the department. When UN Statistics moves from ISIC Rev. 4 to Rev. 5, one person updates the file and every script picks up the new codes on the next run — no copy-pasting revised recodes into five people's .sps files. Teams that want full version history and an audit trail later graduate to hosting the file on a platform like GitHub, but that is a next step, not a prerequisite.

Privacy: the actual labour microdata stays on your laptop. R does not send rows anywhere. Read your existing .sav files with haven::read_sav() — the variable labels and missing-value codes come along intact.
